3. GitHub Workflow
Every module follows a strict branch-based workflow. This ensures main is always clean, each feature is isolated, and rollbacks are easy.
3.1 Branch Strategy
main          →  production-ready code only (never commit directly)
dev           →  integration branch (merge tested features here first)
feature/xxx   →  one branch per module
Examples:
  feature/setup        feature/nav          feature/hero
  feature/about        feature/work         feature/process
  feature/skills       feature/casestudy    feature/testimonials
  feature/contact      feature/polish
3.2 Commit Message Format
Always follow this format for every commit:
feat(section):   short description    ← new feature
fix(section):    short description    ← bug fix
style(section):  short description    ← visual/CSS change
chore:           short description    ← config, setup, tooling
Examples:
  feat(hero): add particle canvas background
  fix(nav): correct mobile menu z-index
  style(about): adjust timeline spacing on mobile
  chore: init project structure and base CSS variables
3.3 Module Workflow (follow every time)
For each module, Claude will follow this exact sequence:
Tell you the branch to create — with the exact git command
Build only that module's code
Give you a test checklist (exactly what to check)
Give you exact git commands to stage, commit, and push
Tell you when and how to merge into dev
Only move to the next module after you confirm it works
3.4 Git Commands Reference
Starting a new module:
git checkout dev
git pull origin dev
git checkout -b feature/hero
Committing your work:
git add .
git commit -m "feat(hero): add particle canvas background"
git push origin feature/hero
Merging into dev after testing:
git checkout dev
git merge feature/hero
git push origin dev
Final merge to main for deployment:
git checkout main
git merge dev
git push origin main
